What is this?

ECO stands for "Eventually Consistent Objects." It is a library for sharing distributed state between users on the Phoenix/SSB network. It provides applications with an API to post object updates onto the feed and merge in updates from other users. Think of it like "Git for object state."

To see how it works, play with the UI above. Each replica represents a different user's view of the same object. Make changes to a replica, commit the changes, and notice how the log is updated. Then hit the sync button: the other replica should adopt the changes and update its log as well.

(Try commiting updates on both replicas, then hit sync to see how concurrent changes are merged.)

Each field in ECO is a different Conflict-Free Replicated Data Type. They are:

  • Counter: a number which combines the increments and decrements of all users.
  • CounterSet: a K/V-map of Counters (not in the demo).
  • Register: a field which takes the most recent write ("Last-Write Wins").
  • Growset: an unordered set which can only add values.
  • Onceset: an unordered set which can add and remove values, but can never re-add a value after it's been removed.
  • Set: an unordered set which can add and remove values freely.
  • Map: a K/V-map of fields which behave like registers (not in the demo).
